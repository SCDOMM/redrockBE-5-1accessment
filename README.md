### 1.服务情况 ###
由于考核要求，项目必须使用RPC写微服务，又由于MQ实在好用，所以就分为了三个包：
1. Order包：用于接收前端传来的下单指令。先用Hertz框架接收请求，随后使用Redis的DECR指令对库存进行扣除，若库存已无，则返回错误；若库存还有，则将请求封装为订单发给RabbitMQ。
2. Storage包：用于接收Order包传来的订单。先从MQ中接收订单，随后将订单发给MySQL进行储存。同时，Storage包使用了Kitex，用于给Check包检查订单状态。
3. Check包：用于给前端提供检查数据库的接口。在订单下单完成后，订单会通过Storage包进入数据库中，而Check包则使用Hertz为前端提供一个接口，前端会反复发送请求，直到Check包通过RPC从Storage包中获取到订单情况为止。

### 2.技术栈 ###
1. Redis数据库：用于进行缓存，限流(暂未实装)等等的非关系型数据库。
2. RabbitMQ和它的动物朋友们：适合分发消息的消息队列及其依赖。
3. Hertz框架：用于进行网络请求
4. yaml v3：用于解析yaml配置文件
5. MySQL：非关系型数据库
6. GORM：对象关系映射，用于以OOP的方式操作数据库
7. 雪花算法：用于给订单赋一个唯一有序的ID
8. Kitex：字节跳动研发的RPC框架
9. Thrift：Kitex使用的IDL

## 二.项目架构 ##
### 1.通用架构 ###
1. router层：用于给Hertz注册路由
2. api层：用于接收前端传来的请求，沟通sv层并将响应，错误等打包发给前端
3. sv层：用于处理业务逻辑，在其中实际处理请求并执行各种逻辑，可能会沟通dao层、cache层或者mq层，或者其他乱七八糟的层
4. utils层：用于存放各种工具函数，比如雪花算法，yaml配置，以及错误打包等等
### 2.特殊架构 ###
1. dao层：用于和MySQL数据库连通，进行增删改查等操作。当然，也可能是借助RPC和其他服务的dao层函数沟通。
2. cache层：用于和redis数据库连通，进行缓存，库存检查，限流等操作。
3. mq层：存放RabbitMQ所必要的一切函数和结构体，用于发送消息或者接收消息。
4. kitex_gen层：使用Kitex时自动生成的层级，包含搞RPC时所必要的一切方法

## 三.方案说明 ##
经过使用俺寻思之力和查各种博客、AI，找迪普西克、糖包和老登学长各种西天取精后，我得出了一套可用(存疑)的秒杀系统方案。

秒杀系统的主要矛盾在于解决高并发，高流量。如果所有服务都是用单线程，搞到数据直接存数据库，那就和光头弄花园口决堤没区别，服务器和数据库会被淹死的。为此，就必须用到多线程和一些额外的妙妙工具。

Redis数据库是一款由Salvatore Sanfilippo开发的意大利产数据库，其可以同时支持50个并发和数万个请求屹立不倒，同时，其的一切操作都是原子性的——这意味着其可以用于一些要求同步的操作。

如果所有操作都是异步的，都是非原子性的，那么就可能出现这样一种情况：A和B同时点了下单按钮，但是库存只剩下了一件商品，那么在A和B同时进行的情况下，就有可能发生库存变为-1的情况——这就是库存超卖问题。

而通过Redis的DECR操作，就可以解决库存超卖问题。再配合Lua脚本，便可以实现更好的操作，把库存超卖问题和非法下单、重复下单问题一并解决。

解决了超卖和重复下单问题还不够，这只是略微减少了洪水的量和速度，这么大的量还是能把数据库淹死。

MessageQueue消息队列是一种异步通信机制，其通过生产者-消费者模型，可以让同一消息同时被多个系统处理。同时，其还能起到缓冲作用，让所有消息被下游消费者慢慢吃掉。

我们可以拿几个服务当生产者，拿另外几个服务当消费者，使用MQ将他们联系起来。这样，一个节点爆炸了还有其他节点可用；同时，由于所有逻辑并不放在同一服务里，服务器的压力也减轻了。

于是，洪水被劈成好几瓣细水，分给了各个服务单独处理。原本能把服务器和数据库送上天的滔天数据流，被Redis和MQ治成傻福了。

消息经过广播和消峰后便可以安全存入数据库了。
